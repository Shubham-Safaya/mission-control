Everyone wants to talk about identity graphs. Nobody wants to talk about identity graph deletion, which is where the actual engineering lives.

Deleting a person from a graph is not deleting a row. Their identifiers thread through merge history, derived audiences, vendor syncs, measurement panels, and backups. True deletion means unwinding a person from a data structure that was specifically built to make people hard to lose.

The pattern that works: treat deletion as a first-class graph operation with its own ledger. Resolve the request to the full identity cluster first, then propagate erasure outward with confirmations logged per system, then tombstone the identifiers so re-ingestion from a stale vendor file cannot resurrect the person next Tuesday.

Resurrection is the failure mode nobody budgets for. A deleted customer who reappears via a quarterly enrichment file is a regulatory finding with a timestamp.

If your deletion design has no tombstones, it is not deletion. It is a nap.
